Findings

F-01Critical

No authorization / RBAC — any authenticated user can do everything

OWASP A01:2021

Every cases route is protected only by @UseGuards(JwtAuthGuard), which proves authentication but never checks authorization. The role field exists on the user model and is carried in the JWT, yet it is never read anywhere in the codebase. A low-privilege staff user can delete any case, delete attachments, and patch any record — identical to admin.

Location
apps/api/src/cases/cases.controller.ts:26,68 · grep for "role" → only set in JWT, never enforced
Evidence
@UseGuards(JwtAuthGuard)        // authentication only
@Controller('cases')
export class CasesController { ...
  @Delete(':id')
  remove(@Param('id', ParseIntPipe) id: number) { ... }   // no role guard
Impact
Privilege escalation by design — destructive operations (case/attachment deletion) are open to the least-privileged account.
Remediation
  • Add a @Roles('admin') decorator + RolesGuard and apply it to destructive routes (DELETE, sensitive PATCH).
  • Define the authorization matrix explicitly; deny by default.
  • Consider per-record ownership/assignee checks if cases should be siloed by team.
F-02Critical

Weak / default JWT secret enables token forgery

OWASP A02:2021

The shipped .env.example sets JWT_SECRET="change-me". config.getOrThrow('JWT_SECRET') validates that a value exists, but not that it is strong or non-default. If the app is deployed with the example value (or any guessable secret), an attacker can forge a valid JWT for any user — and because role is encoded in the token (F-01), forge role:"admin" too. This is a full authentication bypass.

Location
apps/api/.env.example:4 · apps/api/src/auth/auth.module.ts:17 · jwt.strategy.ts:20
Evidence
JWT_SECRET="change-me"   # .env.example — copied verbatim by many deploys
Impact
Forged tokens → complete account takeover and privilege escalation.
Remediation
  • Generate a high-entropy secret per environment (openssl rand -base64 48); never reuse the example.
  • Fail fast on boot if JWT_SECRET equals a known default or is shorter than ~32 bytes.
  • Prefer asymmetric signing (RS256) so the verifier never holds the signing key, if the API fleet grows.
F-03High

Default seeded credentials documented and printed in the UI

OWASP A07:2021

The seed creates admin@forth.com / password123 and staff@forth.com / password123. These exact credentials appear in DEPLOY.md, the README, and are rendered on the production login page. If the seed is run against a real deployment and the password isn't rotated, anyone can log in as admin.

Location
apps/api/prisma/seed.ts:7-8 · apps/web/app/login/page.tsx:90 ("ทดสอบ: admin@forth.com / password123")
Impact
Trivial unauthenticated takeover of a freshly seeded instance.
Remediation
  • Remove the credential hint from the login page in production builds.
  • Force a password change on first admin login, or seed a random password printed only to the deploy log.
  • Keep demo seeds out of production seed paths.
F-04High

No rate limiting or lockout on authentication

OWASP A07:2021

POST /api/auth/login has no throttling, CAPTCHA, or account lockout. Combined with a weak min-length policy (F-11), online brute-force and credential-stuffing attacks run unrestricted.

Location
apps/api/src/auth/auth.controller.ts:20 · no @nestjs/throttler in package.json
Impact
Password guessing at full request rate; no signal or friction for attackers.
Remediation
  • Add @nestjs/throttler with a tight per-IP limit on the login route (e.g. 5–10/min).
  • Add progressive backoff / temporary lockout per account after repeated failures.
  • Ensure the proxy forwards the real client IP for accurate throttling.
F-05High

JWT stored in a JavaScript-readable, non-Secure cookie

OWASP A05/A07:2021

The token is written client-side via document.cookie with only SameSite=Lax — no HttpOnly, no Secure. Any XSS (e.g. F-07) can read and exfiltrate the bearer token, and absent Secure the cookie can travel over plain HTTP.

Location
apps/web/lib/auth.ts:13-15
Evidence
document.cookie = `${TOKEN_COOKIE}=...; path=/; max-age=${MAX_AGE}; samesite=lax`;
// missing: HttpOnly, Secure
Impact
Token theft via XSS; session hijacking; exposure on non-TLS hops.
Remediation
  • Issue the session cookie from the server with HttpOnly; Secure; SameSite=Strict and have the API read it server-side (the Next.js proxy already gates on the cookie).
  • If a JS-readable token is unavoidable, scope it tightly and keep lifetimes short.
F-06High

CORS falls back to wildcard when origin is unset

OWASP A05:2021

origin: process.env.CORS_ORIGIN?.split(',') ?? '*' — if CORS_ORIGIN is missing or empty, the API accepts requests from any origin. A deploy misconfiguration silently opens the API to cross-origin browser calls.

Location
apps/api/src/main.ts:10-12
Impact
Cross-origin abuse of the API from attacker-controlled sites if the env var is ever unset.
Remediation
  • Fail closed: throw on boot if CORS_ORIGIN is missing in production rather than defaulting to *.
  • Maintain an explicit allow-list; never wildcard a credentialed API.
F-07Medium

Upload filter trusts client MIME — SVG enables stored XSS

OWASP A03/A05:2021

The Multer fileFilter accepts anything where file.mimetype.startsWith('image/'). The MIME type is supplied by the client and is spoofable, and image/svg+xml legitimately passes — SVG files can embed <script>. The file is stored in S3 with that content-type and surfaced via a public/presigned URL; opening it directly executes script in the storage origin.

Location
apps/api/src/cases/cases.controller.ts:77-83 · storage.service.ts:58-78
Evidence
if (file.mimetype.startsWith('image/')) cb(null, true);   // client-controlled; allows image/svg+xml
Impact
Stored XSS / phishing hosted on the trusted storage domain; content-type confusion.
Remediation
  • Validate by file signature (magic bytes), not the client MIME; allow-list raster types only (png/jpeg/webp).
  • Block or sanitise SVG; serve attachments with Content-Disposition: attachment and a restrictive CSP.
  • Serve user uploads from a separate, cookie-less domain.
F-08Medium

LINE webhook signature verification skipped when secret unset

OWASP A05/A07:2021

The webhook returns early without verifying the signature when LINE_CHANNEL_SECRET is empty. That's a dev convenience, but a production deploy that forgets the secret silently accepts unauthenticated webhook posts. (The HMAC comparison itself is correctly constant-time via timingSafeEqual — good.)

Location
apps/api/src/notifications/line.controller.ts:52-53
Evidence
// Skip verification when no secret is configured (local/dev only).
if (!this.secret) return;
Impact
Spoofed webhook events and log injection if the secret is unset in production. Impact is currently limited — the handler only logs — but the bypass is a latent risk.
Remediation
  • Require the secret outside development; reject unsigned requests in production.
  • Gate the skip behind an explicit NODE_ENV !== 'production' check.
F-09Medium

No security headers (helmet / CSP / HSTS / frame options)

OWASP A05:2021

The API does not register helmet or set any security headers. There is no Content-Security-Policy, HSTS, X-Content-Type-Options, or frame-ancestors policy as defence-in-depth against MIME sniffing, clickjacking, and XSS.

Location
apps/api/src/main.ts (no helmet) · grep "helmet" → none
Impact
Reduced layered protection; easier exploitation of other client-side issues.
Remediation
  • Add helmet() to the API and a CSP + security headers to the Next.js responses.
  • Enable HSTS at the edge/proxy; set X-Content-Type-Options: nosniff and a restrictive frame policy.
F-10Medium

No token revocation; stale claims trusted for the full lifetime

OWASP A07:2021

JwtStrategy.validate() returns identity straight from the token payload (id, email, name, role) without a DB lookup. There is no server-side revocation — logout only clears the client cookie. A user who is deleted, disabled, or demoted keeps full access (with their old role) until the 1-day token expires.

Location
apps/api/src/auth/jwt.strategy.ts:24-31 · apps/web/lib/auth.ts:17 (client-only logout)
Impact
Revoked/demoted users retain access for up to 24h; no way to force-invalidate a leaked token.
Remediation
  • Re-fetch the user (and current role) from the DB in validate(), or carry a token version / tokenValidAfter timestamp checked per request.
  • Add a revocation/denylist for logout and compromised tokens; shorten access-token lifetime and use refresh tokens.
F-11Low

Weak password policy

OWASP A07:2021

LoginDto enforces only @MinLength(6) and there is no registration-time complexity, breach-list, or length check. Amplifies the brute-force exposure in F-04.

Location
apps/api/src/auth/dto/login.dto.ts:9
Remediation
  • Require ≥12 chars, check against a breached-password list (e.g. HIBP k-anonymity), and apply policy at user-creation time.
F-12Low

No pagination on list / search

OWASP A05:2021

findAll returns every matching case with no limit/offset, and the search term feeds an unbounded OR contains scan. With a large dataset this is a cheap resource-exhaustion vector and a slow, full-table query path.

Location
apps/api/src/cases/cases.service.ts:34-52
Remediation
  • Add mandatory pagination (take/skip or cursor) with a server-side max page size; cap the search input length.
F-13Low

OWASP A09:2021

The LINE webhook logs userId/groupId/roomId for every event, and customer contact details are processed throughout. Logged identifiers are personal data and may be retained/aggregated by the platform.

Location
apps/api/src/notifications/line.controller.ts:44-46
Remediation

F-14Low

SQLite: no at-rest encryption, single writer, backup gap

OWASP A04:2021

Customer complaint data (PII) lives in a plain SQLite file on a single Railway volume. No encryption at rest, limited concurrency, and the deploy notes acknowledge data is wiped without the volume. Acceptable for a demo; risky for production PII.

Location
apps/api/prisma/schema.prisma:5-8 · DEPLOY.md (volume / scaling notes)
Remediation
  • Migrate to managed Postgres with encryption at rest and automated backups before storing real customer data (path already documented in DEPLOY.md).
F-15Low

Attachment deletion is best-effort — orphaned blobs

OWASP A04:2021

removeAttachment swallows storage-delete errors and always removes the DB row. Failed S3 deletes leave orphaned objects that are no longer tracked — a slow data-retention / cost leak and a privacy concern (deleted-from-UI files persist in storage).

Location
apps/api/src/cases/cases.service.ts:187-193
Remediation
  • Record failed deletes for retry (queue / reconciliation job) rather than silently dropping them.
F-16Info

No automated dependency / SCA scanning

OWASP A06:2021

No npm audit / Dependabot / SCA step is present. Dependencies (NestJS 11, Next 16, AWS SDK, Prisma 6) are current at review time, but without continuous scanning, future CVEs go unnoticed.

Remediation
  • Add pnpm audit + Dependabot/Renovate to CI; gate merges on high-severity advisories.

What's already done well

  • No SQL injection — all DB access goes through Prisma with parameterised queries; search uses contains, not string concatenation.
  • Passwords hashed with bcrypt (cost 10) — never stored or logged in plaintext.
  • Input validation — global ValidationPipe with whitelist + forbidNonWhitelisted + transform; DTOs on every write; ParseIntPipe on id params.
  • No user enumeration — login returns one generic message for bad email or password.
  • Constant-time webhook HMAC — LINE signature uses timingSafeEqual with a length pre-check.
  • Update allow-list — UpdateCaseDto restricts mutable fields, preventing mass-assignment.
  • Upload size cap (5 MB) and outbound links use rel="noopener noreferrer".
  • Secrets are git-ignored — only .env.example is tracked; no live credentials committed.
  • Auth enforced server-side — the Next.js proxy gating is cosmetic, with real enforcement in the NestJS guard.

Priority fix order

  1. F-02 + F-01 — set a strong unique JWT_SECRET and add role-based authorization. Together these close the full-takeover path.
  2. F-03 — rotate/remove default credentials and the login-page hint before any real deployment.
  3. F-04, F-05, F-06 — add login throttling, move the token to an HttpOnly+Secure cookie, fail-closed CORS.
  4. F-07 → F-10 — harden uploads, require the webhook secret in prod, add helmet, add token revocation.
  5. F-11 → F-16 — policy, pagination, logging hygiene, Postgres migration, SCA in CI.
Methodology: manual static review of authentication, authorization, input handling, file storage, notification webhooks, session management, and deployment configuration.
